High Videlity — Visualization Data Flow

How audio reaches the visualizer across every playback mode. Generated 2026-05-29, updated 2026-05-30.

There are three orthogonal axes to keep straight when reasoning about how a song ends up animating: audio source (what's making sound), feature source (what's filling frames[]), and clock source (what playbackTime the visualizer indexes into frames with). Each playback mode wires these three differently. Mixing them up is the most common source of "the visualizer is animating but audio is silent" / "audio plays but viz is frozen" bugs.

The visualizer side (constant across modes)

Every render frame, the visualizer reads:

  • frames[ playbackTime * 30 ] → per-time FeatureFrame (FFT-derived loudness / chromagram / onset / bands)
  • stemFeatures?.frame(at: playbackTime) → per-time stem-separated features (drums / bass / vocals / other) when available
  • shazam*Override properties → song-level overrides (BPM, key, danceability, mood, time-sig…) from the external metadata pipeline
  • bandLoudness / bandChromagram / bandOnset (inside the FeatureFrame) → 4-band full-mix split, used when stems aren't available

The visualizer doesn't care how these got populated — that's the per-mode logic below.

Cache hierarchy

Frame cache

Basic audio analysis (FFT / chromagram / onset / loudness / beat).

  • Per-Mac only, ~/Library/Caches/HighVidelity/frames/<sha256>.bin.gz
  • Auto-populated on every uncached play
  • ~900 KB per 5-min song (HVFF v2 format)
  • Looked up first; on miss, frames computed in background and written

Stem cache

Demucs source-separated features (drums / bass / vocals / other). Looked up on every play; populated only by the explicit library-browser batch action.

  • Tier 1 — local SQLite. Sub-100ms read, keyed by shazam-<id> or musicapp-pid-<id>
  • Tier 1b — title+artist alias fallback. Finds rows under hash-<sha> (from batch-caching) or stale pid-keys; aliases to shazam-<id> so subsequent plays hit Tier 1 directly
  • Tier 2 — CloudKit public DB. Cross-user shared cache; populates Tier 1 on hit so the next offline play is instant
  • Tier 3 — fresh Demucs compute. Removed from the play path 2026-05-30. Available only via the Library Browser's "Cache Features" batch action. Demucs runs at ~6× realtime on M1 Pro and saturates Metal hard enough to glitch audio — that tradeoff didn't pencil out during playback
  • Apply-side validator. Before any lookup result lands in stemFeatures, applyStemResult cross-checks the row's stored title against AM's authoritative now-playing title and the cached durationSeconds against AM's catalog duration (4 s slop). Mismatches are rejected and the viz stays on band-split — guards against the alias-bug class of wrong-song serves
  • Maintenance: "Verify stem cache". The Visualizers page exposes an audit that flags rows whose stem bytes don't match their tagged metadata (duplicate-payload-with-different-titles, plus an info-only MusicBrainz duration cross-check) and lets the user delete corrupted rows

Per-song overrides (BPM / key / mood / time-sig) come from a separate lookup chain (external-metadata-pipeline) — more a lookup-with-UserDefaults-memoization wrapping GetSongBPM + AcousticBrainz than a cache in the same shape.

Invariants worth not breaking

Frames must be loaded before the viz can show anything time-aligned. For modes with a pre-analyzed timeline (local + AM), this is frames[] set at song-load. For streaming modes, frames grow 30 Hz from appendLiveFrames.
The playbackTime source MUST match the audio source. Flipping AM to local mid-song without updating which clock is read will index nonsensical frames.
frames is @ObservationIgnored. Reading it from a SwiftUI body causes a 30 Hz invalidation cascade that halves FPS. Use the framesCount mirror for "is anything loaded yet" gating.
stopPlayback() releases the AVAudioPlayer and clears its delegate. Any code path that creates a new player must re-call localQueue.attach(to: player) to re-establish the auto-advance delegate. loadAndPlayLocalFast does this implicitly via startPlayback.
registerPreviewForAlignment is for Shazam-driven preview-catalog matching. Load-bearing for AM / system-audio / mic paths (they need the preview signature to align live audio to pre-analyzed frames). Not load-bearing for local files — they have the full source on disk.
Stem cache LOOKUPS run on play; stem cache POPULATION does not. kickoffStemSeparation and kickoffCloudOnlyStems still fire Tier 1 / Tier 1b / Tier 2 lookups when a track changes — those are cheap and benefit batch-cached songs immediately. They do not call provider.separate(...); that pathway was removed 2026-05-30 and the function bails with [stem] kickoff bail: no cache hit and in-playback compute disabled when every tier misses. Resurrection means restoring the old do/catch at the bottom of kickoffStemSeparation — don't quietly add it back.
Cross-source handoff is symmetric on macOS. Both directions need an explicit teardown of the other source.
Local → AM: every AM entry point flows through kickoffTrackPipeline (playAppleMusicSong; musicKit.play(album:) / play(playlist:) / play(station:); queue auto-advance via musicKit.onTrackChange). That calls ensureMacOSAMPlaybackPath: stops any AVAudioPlayer and engages the system tap on RemotePlayerService with the load-bearing 800 ms delay.
AM → local: loadAndPlayLocalFast calls ensureLocalPlaybackPath: stops ApplicationMusicPlayer (which also clears musicKit.nowPlaying) and flips useSystemAudio off. Without this, startPlayback's guards silently fail.
Don't add a new entry point that bypasses these.

Key files

  • AppModel.swift — loadSong, loadAndPlayLocalFast, startPlayback, stopPlayback, queue ops
  • LocalQueueController.swift — queue state + AVAudioPlayer delegate auto-advance
  • MusicKitController.swift — AM playback, onTrackChange, AM queue
  • SongLoader.swift — load + analyze pipeline + frame cache lookup
  • StreamingAnalyzer.swift — live frame appending for system-audio / mic
  • StemFeatureProvider.swift — Swift bridge to the Demucs sidecar; lookups + alias + audit + delete
  • LibraryBatchCacher.swift — the only call site that triggers fresh Demucs compute, invoked from the Library Browser
  • StemCacheAuditor.swift + StemCacheAuditSheet.swift — "Verify stem cache" maintenance: flags duplicate-payload-different-metadata rows and MB duration mismatches
  • FrameFeatureCache.swift — local frame cache (HVFF v2 format)
  • VisualizerView.swift — the consumer; reads frames / stems / overrides / clock
